# Sepsis Early Detection ML System

Sepsis is a life-threatening condition where **every hour of delay increases mortality risk**.
This project builds a **machine learning pipeline for early sepsis risk detection** using
structured, synthetic clinical data.

This repository was audited and refactored after a serious bug was found in the original
version: the model's headline metrics (ROC-AUC / PR-AUC = 1.0) were **fabricated by target
leakage**, not real performance. See [Known Limitations](#known-limitations) for the full story
— finding and fixing that bug, and being upfront about it, is the point of this refactor.

---

## Problem Statement

Hospitals struggle to:

- Detect high-risk sepsis patients early enough
- Prioritize limited ICU resources
- Reduce sepsis-related mortality and 30-day readmission rates

This project frames it as a supervised ML problem:

> Given a patient's structured clinical features *available at the time of prediction*, predict
> the probability that the patient is septic (`is_sepsis = 1`) early enough for clinicians to act.

## Known Limitations

This section exists because the original version of this repository shipped with a critical bug,
and honesty about it is more valuable than pretending it never happened.

### 1. Target leakage (found and fixed)

The original `train_sepsis.py` dropped only the `is_sepsis` label column from the feature
matrix before training, and left in:

- `diagnosis_Heart Failure`, `diagnosis_Normal`, `diagnosis_Pneumonia`, `diagnosis_Sepsis` — one-hot
  columns encoding the same diagnosis the label is derived from. `diagnosis_Sepsis` is
  **bit-for-bit identical** to `is_sepsis` (`is_sepsis = 1 if diagnosis == "Sepsis" else 0`), so the
  model could trivially "predict" the label from a feature that *is* the label.
- `mortality` and `readmission_30d` — outcomes that only exist **after** the hospital stay
  concludes. A real early-warning system predicting sepsis risk at admission could never have
  these values available; including them is temporally impossible and leaks information a
  clinician-facing model would never have.

With those columns in the feature set, the model reported **ROC-AUC = 1.0 and PR-AUC = 1.0**
(see the previous `reports/sepsis_metrics.json`, preserved in git history) — a perfect score that
should have been an immediate red flag, not a result to report.

**The fix:** `src/sepsis_ml/features.py` is now the single source of truth for feature selection.
It explicitly drops all `diagnosis_*` columns (by prefix, so this holds even if categories
change), plus `mortality` and `readmission_30d`. `src/sepsis_ml/train.py` and
`tests/test_leakage.py` both import this same function, so a regression here would be caught by
CI, not discovered by a suspiciously perfect metric months later.

**Real, post-fix performance** (5x5 repeated stratified cross-validation on
`data/balanced_sepsis.csv`, 18,716 rows, 12 legitimate features):

| Metric | Mean | Std |
|---|---|---|
| ROC-AUC | 0.937 | 0.004 |
| PR-AUC | 0.953 | 0.003 |

Out-of-fold (single stratified 5-fold pass) classification report at the default 0.5 threshold:
accuracy 0.856, sepsis-class precision 0.999, sepsis-class recall 0.713. Full numbers, per-fold
scores, and the exact dropped columns are in `reports/sepsis_metrics.json`, regenerated by
running `sepsis-ml train`. A confusion matrix and precision-recall curve are saved to
`reports/figures/`.

These numbers are **much lower than 1.0, and that's correct** — they reflect real learnable
signal in vitals/labs/comorbidities, not a leaked label.

### 2. Residual leakage from pre-split SMOTE (disclosed, not fixed)

`data/balanced_sepsis.csv` is already balanced 50/50 via SMOTE — the balancing was applied to
the *entire* dataset before any train/validation split existed. That means synthetic minority
samples generated from a given real patient's nearest neighbors can end up on both sides of a
cross-validation fold boundary, letting the model implicitly see interpolated near-duplicates of
its own validation examples. This inflates CV metrics somewhat versus balancing only within each
training fold (the methodologically correct approach).

This repository does not re-architect around this, because the already-balanced CSV is the
reference artifact that shipped and that the metrics above are computed from — SMOTE-within-fold
requires the pre-balance data, which `src/sepsis_ml/data.py` can regenerate (see below) but which
is not bit-identical to the original. If you want the cleaner comparison, generate a fresh
pre-balance dataset with `sepsis-ml data-prep` and wire `imblearn.pipeline.Pipeline` with `SMOTE`
as a CV-fold-scoped step; this is flagged as a natural follow-up, not implemented here, so as not
to conflate "fixed a bug" with "changed the underlying question."

### 3. The original data generation script never existed

The original README described `data/synthetic_clinical_dataset.csv` as the source dataset and a
SMOTE/SMOTENC pipeline as the process that produced `data/balanced_*.csv`. Neither the source CSV
nor any generation code was present anywhere in git history, and `imbalanced-learn` wasn't even a
dependency — the balanced CSVs could not have been reproduced from what was in the repo. See
[Data](#data) below for how this was addressed.

## Data

**The committed `data/balanced_sepsis.csv`, `data/balanced_mortality.csv`, and
`data/balanced_readmission.csv` are treated as the canonical reference artifacts.** All metrics
in this README and in `reports/sepsis_metrics.json` are computed from them, unchanged.

### Reconstructed data generator

Because the original generation code was missing (see Limitation 3), `src/sepsis_ml/data.py`
provides a **documented reconstruction**: a seeded, parametric synthetic-patient generator plus a
SMOTE/SMOTENC balancing step, built to produce data of a similar shape and with similarly
plausible clinical structure (e.g. septic patients trend hypotensive, hyperglycemic, and have
elevated creatinine) to the committed files.

**This is explicitly not claimed to be bit-identical to the original data** — there is no way to
recover the exact original generation logic, since it was never in the repository. Run it with:

```bash
sepsis-ml data-prep --n-patients 10000
```

By default this writes to `data/reconstructed/` (gitignored) and does **not** touch the committed
`data/*.csv` files. Pass `--overwrite-canonical` only if you deliberately want to replace the
reference data (not recommended — the metrics reported here would no longer describe the file
that produced them until you retrain and update this README).

### Columns

- **Demographics:** `age`, `sex`, `bmi`
- **Vitals / Labs:** `systolic_bp`, `diastolic_bp`, `glucose`, `cholesterol`, `creatinine`
- **Comorbidities:** `diabetes`, `hypertension`
- **Diagnosis:** `diagnosis` in {`Normal`, `Pneumonia`, `Heart Failure`, `Sepsis`}
- **Outcomes (post-admission, never used as model features):** `mortality`, `readmission_30d`

`data/balanced_sepsis.csv` is the numeric + one-hot-encoded, SMOTE-balanced (50/50 on
`is_sepsis`) version used for training. `data/balanced_mortality.csv` /
`data/balanced_readmission.csv` are raw-format, SMOTENC-balanced datasets for the (not yet built)
mortality/readmission models — see [Roadmap](#roadmap).

> These datasets are synthetic and intended for portfolio / pipeline-demonstration purposes.
> No real patient data is used; no PHI/PII is present.

## Model & Training Pipeline

**Target:** `is_sepsis` in {0, 1}
**Model:** `GradientBoostingClassifier` (scikit-learn), `n_estimators=200`,
`learning_rate=0.05`, `max_depth=3`, `random_state=42`, behind a `StandardScaler`.

**Pipeline** (`src/sepsis_ml/train.py`):

1. Load `data/balanced_sepsis.csv`.
2. Build a leakage-free feature matrix via `sepsis_ml.features.prepare_features` (drops
   `diagnosis_*`, `mortality`, `readmission_30d`; see Known Limitations).
3. Evaluate with **5x5 repeated stratified k-fold cross-validation** (`RepeatedStratifiedKFold`)
   for headline ROC-AUC / PR-AUC, plus a plain stratified 5-fold pass to produce a single set of
   out-of-fold predictions for the confusion matrix and PR curve.
4. Fit a final model on all available data and persist it.
5. Save metrics (`reports/sepsis_metrics.json`) and plots
   (`reports/figures/confusion_matrix.png`, `reports/figures/pr_curve.png`).

## How to Run Locally

### 1. Clone and install

```bash
git clone https://github.com/<your-username>/sepsis-early-detection-ml.git
cd sepsis-early-detection-ml
python3 -m venv .venv
source .venv/bin/activate  # Windows: .venv\Scripts\activate
pip install --upgrade pip
pip install ".[dev]"
```

(`requirements.txt` is also kept in sync for environments that prefer `pip install -r
requirements.txt`.)

### 2. Train the model

```bash
sepsis-ml train
# or: python -m sepsis_ml.train
```

Optional flags: `--data-path`, `--models-dir`, `--reports-dir`, `--n-splits`, `--n-repeats`,
`--random-state`. Run `sepsis-ml train --help` for the full list.

### 3. (Optional) Regenerate a reconstructed dataset

```bash
sepsis-ml data-prep --n-patients 10000
```

### 4. Run tests

```bash
pytest -q
```

### 5. Docker

```bash
docker build -t sepsis-ml .
docker run --rm -v "$(pwd)/data:/app/data" -v "$(pwd)/models:/app/models" \
    -v "$(pwd)/reports:/app/reports" sepsis-ml
```

## Governance, Risk & Compliance Notes

Although this project uses synthetic, non-PHI data, it is designed in a way that could be safely
extended toward a real setting:

- **No PHI/PII:** no names, MRNs, dates of birth, addresses, or identifiers.
- **Data minimization & explainability:** structured features (vitals, labs, comorbidities) that
  can be clinically interpreted and audited.
- **Imbalance handling:** SMOTE/SMOTENC balancing, documented and — per Limitation 2 above —
  honestly flagged as applied before rather than within CV folds in the shipped reference data.
- **Leakage discipline:** feature selection is centralized and covered by a regression test
  specifically because a leakage bug shipped once already.

In a real deployment you would additionally need: access control & audit trails, PHI scrubbing
before logging, model monitoring (drift, performance degradation), and bias checks across
age/sex/comorbidity slices. None of that is implemented here — this is a portfolio-scale
demonstration, not a deployable clinical system.

— this list is the honest state as of this refactor.

## Why This Project Matters

This project demonstrates the ability to:

- Take a messy clinical prediction problem (sepsis) and frame it as supervised ML.
- **Audit your own pipeline critically enough to find a target-leakage bug that produced a
  fabricated perfect score, fix it, and disclose it** — rather than shipping the 1.0 AUC.
- Design a balanced, imbalance-aware training dataset with a documented (if reconstructed) data
  pipeline.
- Package a model into a reproducible, tested, CI-checked pipeline with saved artifacts.
- Reason about governance, compliance, and methodological limitations, not just write code.
